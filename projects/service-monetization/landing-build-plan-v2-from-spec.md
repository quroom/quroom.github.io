# Landing 구축 실행계획 v2 (quroom-landing-spec 반영)

기준 문서:
- `landing/openspec/changes/consolidate-codex-docs-into-openspec/docs/quroom-landing-spec.md`

## 1) 프로젝트 목표 재정의 (Spec 기준)
- 사이트 형태: 원페이지
- Primary CTA: 상담 문의 접수
- Secondary CTA: 서비스 소개서 요청 / 협업 제안
- 핵심 메시지 2축:
  1. 아이디어→서비스→수익 연결(실행형 제품 개발)
  2. 외국인 개발자 실무 연계 지원

## 2) 서비스 확장 목표(요청 반영)
Spec의 2축을 유지하면서, 서비스 확장 블록을 추가:
- 자동화 세팅(OpenClaw 구축 포함)
- 바이브코딩 코칭
- 개발 외주(고객 맞춤 구축)
- 향후 해외 개발자 연계 에이전시 모델

> 원칙: 기존 Spec 톤(전문적/신뢰/실행 중심)을 깨지 않고 확장 서비스는 "추가 Offer"로 배치

## 3) 정보구조(최종)
1. Header (고정형 + CTA)
2. Hero (핵심 가치 + 동적 연차/프로젝트 신뢰 문구)
3. About (회사 소개 + 핵심 지표)
4. Services
   - Spec 서비스 2개(핵심)
   - 확장 Offer 3개(자동화/바이브코딩/OpenClaw)
5. Portfolio (문제-실행-결과 구조)
6. Contact (폼 + 이메일 + 응답시간)
7. Footer (법정 정보/정책 링크)

## 4) 구현 백로그 (2주)

### Day 1~2: 콘텐츠 동기화
- Spec의 기업 정보/연차 계산 로직 반영
- Hero/About/Footer 문구 일관성 체크
- 프로젝트 수 표기 고정: `총 7건 (개인 5, 외주 2)`

### Day 3~4: 섹션 구현
- Django + htmx + Tailwind 원페이지 레이아웃 구현
- 고정 헤더 + 섹션 앵커 이동
- Hero 배경/CTA 배치

### Day 5: 서비스 섹션 확장
- Spec 2개 핵심 서비스 카드 구현
- 확장 Offer(자동화/바이브코딩/OpenClaw) 보조 카드 추가

### Day 6~7: 포트폴리오
- 카드 템플릿(문제-해결-결과) 적용
- 공개/익명 프로젝트 혼합 노출
- 썸네일 경로 규칙 검증

### Day 8: 문의 폼
- Django Form + htmx POST
- 자동응답 메일
- 개인정보 동의/문의 유형 필드 반영

### Day 9: SEO/OG/파비콘
- title/description/OG/meta 키워드 반영
- 대표자 SNS 링크 라벨(개인 계정) 검증

### Day 10: 품질/배포
- Lighthouse 목표 점검
- 반응형(모바일/태블릿/데스크탑) QA
- 배포 및 폼 동작 실검증

## 5) 외주/해외연계 에이전시 확장 계획

### 단계 1: 현재 랜딩 전환 안정화
- 상담 전환 데이터 2~4주 수집
- 어떤 오퍼(자동화/코칭/외주)가 클릭/문의 많은지 파악

### 단계 2: 에이전시형 오퍼 런칭
- 고액 외주 패키지 3단계(Lite/Standard/Pro)
- PM(한국) + Dev(해외) 역할 정의
- 변경요청/추가비용/커뮤니케이션 규칙 문서화

### 단계 3: 다국어 확장
- KR 우선, EN 보조 페이지
- 해외 개발자 지원/협업 안내 섹션 분리

## 6) 반드시 지킬 검증 항목 (Spec 매핑)
- 기업/대표/주소/사업자 정보 일치
- LinkedIn/Threads 라벨: `대표자 개인`
- 연차 동적 계산 로직 표기 일치
- CTA 동작/문의 폼 동작 확인
- 정책 링크(/privacy, /terms) 유효

## 7) 이번 주 의사결정 포인트
1. 확장 Offer를 Hero에 노출할지, Services 하단에만 둘지
2. 외주 가격 범위를 공개할지(문의형으로 둘지)
3. Contact CTA를 단일화할지(상담/협업 분리 유지)
