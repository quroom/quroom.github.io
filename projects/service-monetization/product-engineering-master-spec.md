# Product Engineering Master Spec (기획·개발·운영·마케팅 통합 실행본)

## 0. 전체 목차 + 문서 의존관계

### 0-1. 실행 순서(의존관계)
1. **PRD** → 문제/가설/범위 고정
2. **IA & User Flow** → 화면/행동 단위 확정
3. **기능 우선순위(MoSCoW)** → 개발 범위 컷오프
4. **데이터 스키마 + API 경계** → 구현/운영 기준선
5. **개발 백로그 + QA 플랜** → 일정/품질 실행
6. **운영 런북** → 장애/문의/배포 대응 체계
7. **마케팅·세일즈 플랜** → 리드/전환 실행
8. **KPI 대시보드 + 30-60-90 로드맵** → 성장/의사결정 체계

### 0-2. 가장 레버리지가 큰 문서 3개 (우선 작성)
- 1순위: **PRD**
- 2순위: **개발 백로그**
- 3순위: **운영 런북**

### 0-3. 가정
- 역할: 1인 Product Engineer
- 우선순위: 사용자 수보다 **매출/전환** 우선
- 목표: 90일 내 10M~20M KRW 외주 1건 수주
- 채널: LinkedIn, Threads, 지역 네트워크(광주/전라도권)

## 1. PRD (문제/가설/범위/리스크/지표)

### 1-1. 목표
- 4주 내 유료 고객 2건 이상 확보
- 90일 내 고액 외주 제안서 5건 발송, 계약 1건 체결

### 1-2. 문제 정의
- 잠재 고객은 “개발+자동화”를 듣고도 **무엇을 언제 받는지** 즉시 이해하지 못함
- 상담 이후 결제까지 전환 흐름이 표준화되어 있지 않음
- 오퍼/사례/후속 루프가 문서 단위로 흩어져 실행 속도가 느려짐

### 1-3. 핵심 가설
- G1. 오퍼 패키지를 3단(90분/2주/4~8주)으로 고정하면 상담→결제 전환율이 상승한다.
- G2. 상담 후 24시간 내 요약+다음액션 제공 시 응답률이 30% 이상 증가한다.
- G3. 사례 카드(문제-실행-결과)를 제안서에 포함하면 고액 오퍼 수락률이 증가한다.

### 1-4. 범위(In Scope)
- 오퍼 카탈로그 3종 확정
- 랜딩/상담/제안/후속 플로우 문서 표준화
- 리드 파이프라인 대시보드 정의 및 주간 점검
- 2~4주 마케팅 실행 캘린더 운영

### 1-5. 비범위(Out of Scope)
- 투자 자문/수익보장형 컨설팅
- 다국어 전체 로컬라이제이션
- 대규모 팀 협업 기능 개발

### 1-6. 리스크 및 대응
- R1. 리드 부족 → 기존 네트워크 20명 직접 아웃바운드
- R2. 제안서 미응답 → D+1, D+3, D+7 후속 표준 실행
- R3. 범위 확장 요구 → SOW 변경관리 규칙 적용

### 1-7. 핵심 지표
- Lead→Consult 전환율
- Consult→Paid 전환율
- 제안서 발송 수/수락률
- 건당 마진율, 시간당 수익

### 1-8. 체크리스트
- [ ] 오퍼 3종 가격/산출물 고정
- [ ] 상담 스크립트 최신화
- [ ] 제안서 템플릿에 사례 카드 반영
- [ ] 후속 접촉 자동 체크리스트 실행

### 1-9. 즉시 실행 TODO
1. 이번 주 제안서 템플릿 1종 최종 고정
2. 상담 후 발송용 요약 템플릿 작성
3. 최근 3개 사례를 카드 포맷으로 변환

### 1-10. 검토 질문 3개
1. 이번 사이클의 “성공”을 매출 기준으로 얼마로 볼 것인가?
2. 90분/2주/외주 중 현재 최우선 판매 오퍼는 무엇인가?
3. 계약 실패의 공통 원인은 무엇이며 문서에서 어떻게 예방할 것인가?

## 2. IA & User Flow

### 2-1. 목표
- 유입~결제까지 사용자의 다음 행동을 1클릭 수준으로 단순화

### 2-2. 정보구조(IA)
- 홈: 가치제안 + 대표 사례 + CTA
- 오퍼: 3개 패키지 비교
- 사례: 전/후 지표 기반 카드
- 상담예약: 일정/문제/예산 입력
- 후속페이지: 제안서/결제/온보딩

### 2-3. 핵심 사용자 여정
1. 콘텐츠 유입(Threads/LinkedIn)
2. 랜딩 방문
3. 오퍼 확인 + 사례 열람
4. 상담 예약
5. 상담 진행
6. 제안서 수신
7. 결제/온보딩

### 2-4. 주요 화면
- 랜딩(신뢰+문제공감)
- 오퍼 비교표
- 상담 신청 폼
- 상담 후 제안서 뷰
- 결제/착수 안내

### 2-5. 체크리스트
- [ ] 각 화면 CTA 1개로 제한
- [ ] 사례 카드에 수치 포함
- [ ] 예약 폼 질문 5개 이내

### 2-6. 즉시 실행 TODO
- 랜딩 상단 3초 메시지 문구 확정
- 오퍼 비교표(가격/기간/산출물) 고정
- 상담 폼 필수 문항 정리

### 2-7. 검토 질문 3개
1. 유입 채널별 랜딩 메시지를 분리할 필요가 있는가?
2. 상담 신청 폼에서 꼭 받아야 하는 최소 정보는 무엇인가?
3. 사례 페이지 없이도 전환이 가능한가?

## 3. 기능 우선순위표 (MoSCoW)

### 3-1. Must
- 오퍼 3종 비교표
- 상담 신청/접수
- 제안서 템플릿
- 후속 접촉 시나리오

### 3-2. Should
- 사례 카드 라이브러리
- FAQ 자동 확장
- 주간 KPI 리포트

### 3-3. Could
- 외국어 버전 랜딩
- 제안서 자동 생성 도구

### 3-4. Won’t (이번 사이클 제외)
- 대규모 SaaS 제품화
- 팀 권한 관리 시스템

### 3-5. 체크리스트
- [ ] Must 항목만으로 매출 발생 가능한지 검증
- [ ] Should는 시간 여유 시 착수

### 3-6. 즉시 실행 TODO
- Must 항목 담당/마감일 부여

### 3-7. 검토 질문 3개
1. Must 항목 중 실제 매출 기여도가 가장 큰 것은?
2. Should를 이번 주에 착수하면 무엇이 밀리는가?
3. Won’t 목록을 팀/협업자에게 명확히 공유했는가?

## 6. 개발 백로그 (상세)

### 6-1. Epic A: 전환형 랜딩
- Story A1: 오퍼 비교표 구현
  - AC: 3개 패키지 가격/기간/산출물 노출
- Story A2: 사례 카드 블록 추가
  - AC: 최소 3개 사례, 각 사례에 수치 1개 이상

### 6-2. Epic B: 상담→제안→결제
- Story B1: 상담 신청 폼
  - AC: 필수 5문항, 제출 후 확인 메시지
- Story B2: 제안서 템플릿 자동화

- Story D1: 주간 퍼널 리포트
  - AC: Lead→Consult→Deal 수치 자동 집계
